# Hyperliquid MEV: A Research Dossier

*Educational reference for a Solana MEV course. Figures carry as-of dates where available; numbers drawn from vendor or secondary sources are flagged as such. The throughline: Hyperliquid does not eliminate MEV — it replaces Solana/Jito's discretionary, auctioned MEV economy with deterministic in-protocol ordering, leaving mainly latency- and cross-venue-based extraction, and socializing the largest "MEV" stream (backstop liquidations) into a community vault.*

---

## 1. Hyperliquid Architecture

**HyperBFT consensus.** Hyperliquid's official docs describe HyperBFT as "a variant of HotStuff consensus," optimized for end-to-end latency. Blocks are produced by validators in proportion to staked HYPE. Finality is deterministic and single-block (no reorgs), with optimistic responsiveness finalizing blocks in ~0.07s (~70ms) under normal load (p99 below ~0.5s). It tolerates up to *f* Byzantine validators out of 3f+1 (>2/3 honest required). The "pipelined" property (leader proposes the next block while the prior is still being voted on) is a general HotStuff-family trait, not a verbatim Hyperliquid claim. The MEV-resistance basis is sound: consensus produces one canonical finalized sequence with no competing execution traces.

**HyperCore vs HyperEVM.** HyperCore is the native L1 holding margin and matching-engine state; it runs a fully on-chain central limit order book (CLOB) that "does not rely on the crutch of off-chain order books." HyperEVM is a separate general-purpose EVM (Chain ID 999 mainnet / 998 testnet; Cancun hardfork without blobs; EIP-1559) sharing the same HyperBFT block stream. HyperEVM reaches HyperCore via the **CoreWriter** system contract at `0x3333...3333`. All core financial state transitions anchor to the single HyperBFT-finalized block sequence. The MEV-relevant matching logic lives in HyperCore, not in user-orderable EVM calls.

**No public mempool of pending orders.** Hyperliquid does not expose a public gossip mempool of pending trading actions; HyperCore orders are sequenced by HyperBFT before execution. (HyperEVM does maintain on-chain mempools — constrained to the next 8 nonces per address, transactions older than 1 day pruned — but these are the EVM layer, not the CLOB.) This blocks classic pending-tx sandwiching of a visible victim order but does **not** eliminate latency arbitrage: a co-located trader still reacts fastest to public market data, and the HyperBFT leader still orders the raw incoming stream.

**Deterministic intra-block ordering.** Within a consensus batch, actions are bucketed into three sequential categories, executed in order: (1) actions that send no GTC/IOC order to any book, (2) cancellations, (3) actions sending at least one GTC/IOC order. Within each category, actions keep their original proposal order. This removes *discretionary* validator/searcher reordering and the Jito-style tip-for-ordering economy — but does **not** remove all MEV. Deterministic, publicly known rules (e.g. cancel-before-place) are themselves gameable, and latency races persist (per QuillAudits, deterministic action ordering "still creates MEV-like opportunities").

**CoreWriter delay.** Order actions and vault transfers sent from CoreWriter "are delayed onchain for a few seconds" to "prevent any potential latency advantages for using HyperEVM to bypass the L1 mempool." (The docs deliberately say "a few seconds" — no precise figure.) CoreWriter burns ~25,000 gas before emitting a log; a basic call uses ~47,000 gas total. Read precompiles deployed to mainnet April 2025; CoreWriter (write path) activated July 5, 2025.

**Validator set / block timing.** A small set — roughly two dozen active validators as of mid-2026, up from ~16 at the April 2025 launch — with a planned increase from 24 to 27. The active set is the top accounts by staked HYPE; validation is permissionless and influence is proportional to stake (self + delegated; 10,000 HYPE self-delegation locked for one year). Matching *order* is deterministic, but the small, partly foundation-influenced set still controls which raw transactions land in which block — a residual ordering/trust surface. (Exact count is a moving target across sources; degree of foundation control is genuinely contested.)

## 2. Order & Matching Model

**On-chain CLOB with price-time priority.** HyperCore maintains a per-asset order book that "works similarly to that of centralized exchanges," matched in strict price-time priority (best price first; FIFO within a price level). Price must be an integer multiple of tick size; size an integer multiple of lot size. Margin is checked when a new order opens and again on the resting side at each match. This is a genuine CME/Nasdaq-style execution model implemented in L1 consensus state — unusual for a fully on-chain venue. Capacity: ~200,000 orders/sec; end-to-end latency median ~0.2s, p99 ~0.9s for co-located clients (official HyperCore overview).

**Order & TIF types.** TIF: **GTC** (rests until filled/cancelled), **IOC** (cancelled if not immediately filled), **ALO/Post-Only** (added as a resting order only; cancelled if it would cross — guarantees maker status). Order types: Market, Limit, Stop Market, Stop Limit, Take Market / Take Limit, **Scale** (multiple limits across a range), **TWAP** (parent split into 30-second suborders, each catch-up suborder capped at 3x normal size, 3% per-suborder max slippage), and **Reduce-Only**. Stops and TP/SL trigger on the composite **mark** price, not the HyperCore last trade — defeating single-print stop-hunt MEV.

**Oracle pricing (push model).** Each validator publishes a spot oracle price for every perp asset **every 3 seconds**, computed as a weighted median of eight CEX spot mids: Binance (3), OKX (2), Bybit (2), Kraken (1), KuCoin (1), Gate.io (1), MEXC (1), Hyperliquid (1). The clearinghouse's final oracle price is the **stake-weighted median** of all validator submissions. Asset-specific exclusions: HYPE-style assets (primary liquidity on HL) exclude external sources until liquidity suffices; BTC-style assets exclude the Hyperliquid spot input. No single party can arbitrarily override the price.

**Mark price (robust, multi-input).** Mark price = median of: (1) oracle price + a 150-second EMA of (HL mid − oracle), (2) median of HL best bid / best ask / last trade, (3) weighted median of CEX *perp* mids — Binance (3), OKX (2), Bybit (2), Gate.io (1), MEXC (1). Fallback: with only two of three inputs, a 30-second EMA of the median of HL bid/ask/last is added. Because liquidations and stops fire on this multi-venue mark — not a single HyperCore print — an attacker cannot trigger liquidations by pushing one venue's last trade.

**Funding.** Hourly, premium-driven. F = average Premium Index P + clamp(interest_rate − P, −0.0005, 0.0005). Interest component = 0.01%/8h = 0.00125%/hour. Premium = impact_price_difference / oracle_price, sampled every 5 seconds and averaged over the hour. The computed rate is an 8-hour figure, paid hourly at one-eighth, **capped at 4%/hour**. Payment = position_size × oracle_price × funding_rate. Impact prices plus time-averaging make funding manipulation expensive and short-lived.

## 3. MEV Taxonomy on Hyperliquid

For each: **who captures it**, **profitability**, and **whether the analogous Solana strategy works**.

**Backstop liquidations — the dominant internalized stream.** Most liquidations are first sent as market orders to the book, where anyone can compete and the liquidated user keeps remaining margin. If equity falls below 2/3 of maintenance margin without filling, a **backstop liquidation** transfers the position to the Liquidator Vault (a component of HLP) and the maintenance margin is **not** returned (giving the vault a buffer). Docs: "the pnl stream from liquidations go entirely to the community through HLP." *Who captures:* HLP depositors (community), not external searchers. *Profitability:* profitable on average but path-dependent — HLP also wears tail losses. *Solana analog:* On Solana, lending/perp liquidations (Kamino, MarginFi, Drift, Solend) are permissionless races won by the fastest bot, and the bonus accrues to that external liquidator. Hyperliquid socializes this instead. Mechanics: maintenance margin = half the initial margin at max leverage (leverage ranges 3x–40x), i.e. ~1.25% (40x) to ~16.7% (3x). For positions over 100k USDC, only 20% is liquidated initially, then a 30-second cooldown before full-position orders resume.

**Oracle-update / funding arbitrage.** Validators publish spot oracle prices every 3s; low open interest and few funding-arb market makers have repeatedly let large parties skew HL funding away from CEXes. *Who captures:* co-located/timing-advantaged traders. *Profitability:* real, especially on lower-liquidity perps. *Solana analog:* Solana perps read pull oracles (Pyth, Switchboard) on demand per-program; there is no single consensus-level robust index to arb against in the same way.

**Cross-venue (HL vs CEX) funding/basis arbitrage — most accessible external strategy.** HL base fees (0.015% maker / 0.045% taker) undercut Binance USDT-M standard (0.020% / 0.050%) on both legs, improving arb economics. *Who captures:* external HFT/quant desks, not the protocol. *Profitability:* BTC/ETH spreads have compressed toward break-even; long-tail wider. (Specific return figures — ~3–12% net APR on majors, 20–60%+ long-tail, ~1.3 bps break-even per 8h — come from a single arb-desk blog; treat as illustrative, not hard fact.) *Solana analog:* same basis/funding arb exists across Solana perps venues vs CEX; this is the most portable strategy.

**JIT liquidity / HLP as universal counterparty.** HLP runs multiple market-making strategies plus the Liquidator Vault, acts as counterparty of last resort when the book is thin, and accrues a share of trading fees. *Who captures:* socialized to HLP depositors (vs auctioned to searchers elsewhere). *Profitability:* TVL ~$300–400M (Oct 2025), crossing $500M later in 2025; ~15–30% APR across quarterly windows with 5–12% drawdowns in fast directional moves (independent risk analyses — historical estimates, not official). *Solana analog:* JIT-LP MEV on Solana DEXs/perps pays external LPs/keepers; Drift's perps DEX is the closest analog but its JIT auction still pays external makers, not one protocol-owned vault.

**HIP-3 builder-deployed perps.** Reached mainnet **October 13, 2025**. HyperCore still handles execution, margin, liquidations, and the book, but the builder/deployer **selects and manages the price oracle** that sets mark price for liquidations and funding — concentrating manipulation/MEV risk in the deployer's oracle choice. Staking requirement: 500k HYPE (slashable via stake-weighted validator vote; expected to decrease over time). *Adoption:* builder-market OI grew rapidly past $1B within months of launch (tokenized stocks/commodities leading). *Solana analog:* permissionless market deployment with deployer-chosen oracles carries the same "trust the deployer's oracle" risk.

**Classic sandwiching — does NOT map onto HyperCore.** No public gas-priority mempool of pending taker orders; consensus is action-aware (non-GTC/IOC → cancels → GTC/IOC) with strict price-time priority under one-block finality. The EVM/Solana "observe-then-bracket-the-victim" pattern is structurally foreclosed on the CLOB. **Caveat:** within categories, ordering is still the block proposer's, so proposer-level ordering power is non-zero — a debated fairness point. This is structural/economic resistance, not mathematical impossibility; co-located actors still compete on latency.

**HyperEVM — Solana-like open MEV (smaller).** The separate EVM layer orders txs by gasPrice high-to-low then nonce, with priority fees **burned**, so bots adjust gasPrice to front/back-run. ~2-second small blocks and ~2-second windows between HyperEVM DEX prices and HyperCore spot create arbitrage. A documented two-brother team reportedly netted ~$5M over ~6–8 months on ~$12.5B volume (against ~$1.2M gas; staked ~100k HYPE for a 30% rebate; lowered their profit threshold from 0.15% to 0.05%). *Present the ~$5M/$12.5B figure as a corroborated-but-secondary anecdote, and the timeframe as "~6–8 months."* CoreWriter order/vault actions from the EVM are delayed a few seconds as explicit anti-latency-MEV mitigation.

## 4. Order Flow & Sequencing

**Path: user → API/validator → matching engine.** A user signs an EIP-712 action (via an agent/API wallet) and POSTs it to `/exchange` (queries go to `/info`). The order travels to validators **without entering a public mempool**; HyperBFT sequences it immediately, and the matching engine **is part of the state-transition function** — there is no off-chain sequencer or matcher ("HyperCore does not rely on the crutch of off-chain order books"). Validators run consensus **and** the matching engine natively. Non-validating nodes are permissionless relays/data feeds; an API server points at a local non-validating node for low-latency fills. **There is no separate block-builder role** — validators both order and execute. The Foundation runs a public non-validating node in AWS apne1-az1 (no availability/performance guarantee).

**Builder codes ≠ block builders.** A "builder code" is a native fee-sharing tag for front-ends/apps, attached as `{"b": address, "f": number}` (f in tenths of a bp; f=10 → 1bp). Max fee 0.1% perps / 1% spot (spot charged only on the quote side). `ApproveBuilderFee` must be signed by the user's **main** wallet; max 10 active approvals per user; a builder needs ≥100 USDC perps account value and "standard" account-abstraction mode. **Builder codes attach a fee/routing tag only and do not affect matching or ordering.** As a revenue channel they are large: >$40M cumulative to developers and ~40% of DAU via third-party front-ends (mid-2026; figures approximate/secondary). Leaderboard (mid-2026): Phantom ~$20.6M cumulative since its July 2025 launch (~$37B routed, >$100k/day); BasedOneX ~$15.1M; PVP.trade ~$7.9M lifetime.

**No Jito-style bundle auction.** There is no off-chain block engine, no sealed-bid bundle auction, no top-of-block market, no atomic multi-tx bundles sold to a leader. Ordering is deterministic in-protocol via HyperBFT.

**Native priority fees (April 13, 2026, alpha) — burned, not auctioned.** Hyperliquid *did* add an in-protocol priority lever, but it is fundamentally unlike Jito's: fees are **burned**, not paid to validators/stakers. Two systems:
- *Gossip priority (read-side):* two independent Dutch-auction slots on a synced 3-minute schedule; bid via `gossipPriorityBid` (maxGas in wei, 1e-8 HYPE); each auction resets to 10x the prior winning price with a 0.1 HYPE floor; ~25ms latency reduction per slot (empirical, alpha).
- *Order priority (write-side):* param `{"p": rate}`, rate = p/1e8, charged as a fraction of filled notional in HYPE from undelegated staking balance; **cap 8 bps** (reduced from an initial 20 bps on user feedback); `effective_time = arrival_time + f(action, priority_fee)`, strictly decreasing in fee; **all cancels are prioritized ahead of all immediately-executable orders** (f=0 for cancels); supported only when every order in the action is **IOC on a non-outcome asset**. ~45ms latency reduction per 1bp (empirical, alpha). Burned fees visible via `priorityGas` in user_fills.

So: avoid claiming "zero priority mechanism" — the correct framing is a **constrained, burn-funded, in-protocol fee**, not a validator-revenue bundle auction.

**Rate limits (dual).** IP/weight-based for requests (1200 aggregated weight/minute; exchange action weight = 1 + floor(batch_length/40); WS: 10 connections, 1000 subscriptions, 2000 messages/min, etc.) **and** address-based tied to cumulative volume (1 request per 1 USDC cumulative volume, 10,000 initial buffer; open-order cap 1000 + 1 per 5M USDC, capped at 5000). A batched order is 1 request for IP limits but *n* for address limits. Note: base request capacity is *earned by trading volume*, not bid for.

## 5. Infrastructure & Latency Edge

**Fixed Tokyo hot region.** Glassnode research (covered by CoinDesk, 2026-03-30) measures Hyperliquid's validators clustered in **AWS ap-northeast-1 (Tokyo)** across multiple AZs, with the API fronted by CloudFront. (Hyperliquid has not officially published the region; it is measured/inferred.) Active set: the top-21 by stake (Glassnode/CoinDesk reported ~24 clustered in Tokyo; registered nodes can exceed the active set). Use "~21–24 validators, all in AWS Tokyo."

**The geographic edge — and why it's bounded.** Tokyo-colocated clients reach validators in **~2–3 ms** vs **>200 ms** for European clients — roughly a **200 ms** geographic edge. But measured AWS-Tokyo order-to-fill median was **884 ms** (~879 ms server/consensus-side, only ~5 ms network) vs ~1,079 ms from Ashburn, VA. **The overwhelming majority of total latency (~880 ms) is HyperBFT consensus/server time, not network** — so colocation only trims the small network tail and cannot beat block time. This is the structural cap on the infra arms race.

**Read-side acceleration is possible; the write path is not.** Third parties can speed *market-data reads*, but every order is submitted through the same public execution infrastructure that no provider controls. A vendor "Order Book Server" benchmark (Dwellir, 2,662 trades) showed a **51 ms (24.1%) median improvement** over the public WebSocket, faster in 75.5% of cases, with fewer tail spikes (vendor-reported). Tail-latency reduction matters more than the median for inventory-managing market makers.

**Infra ladder (durable concept; specific prices/throughput are vendor claims):** public RPC (rate-limited, prototyping) → dedicated managed RPC → gRPC streaming → dedicated/colocated non-validating node (`hl-visor run-non-validator`) peered for lowest-latency raw reads. Recommended node hardware: ≥32 logical cores, ≥500 MB/s disk, ~128 GB RAM, Ubuntu 24.04 only; network generates ~100 GB/day of data (pruning/archival required).

**Consensus sets the floor.** ~0.2s median finality, <0.9s p99 (co-located, Hyperliquid benchmark); ~0.07s optimistic responsiveness under normal load. *Teach honestly:* independent third-party measurement (AWS Tokyo, March 2026, ~120 samples) saw median ~884 ms under real conditions — present ~0.2s as best-case/benchmark, not guaranteed live latency.

## 6. Economics

**Trading fees.** Base perp: **0.045% taker / 0.015% maker**. Base spot: **0.070% taker / 0.040% maker**. Six volume tiers by 14-day rolling volume (>5M … >7B); top tier perps 0.024% taker / 0.000% maker, spot 0.025% / 0.000%. Maker rebate tiers down to −0.003% (>3.0% maker share). Spot volume counts double toward the tier.

**HYPE staking discounts (distinct from validator emissions).** Six trading-fee rebate tiers: Wood 5% (>10 HYPE), Bronze 10% (>100), Silver 15% (>1,000), Gold 20% (>10,000), Platinum 30% (>100,000), Diamond 40% (>500,000).

**Fees go entirely to the community — no team cut.** Docs: "fees are entirely directed to the community (HLP, the assistance fund, and deployers)." Spot and HIP-3 deployers may retain up to 50% of trading fees on their markets (configurable). No VC/team/insider fee take.

**Assistance Fund buybacks.** ~97% of protocol trading fees fund automated on-chain open-market **HYPE buybacks**; a Dec 2025 governance vote (passed by ~85% of validators) raised allocation to 99% for certain fee categories plus permanent burns. 2025 scale: Hyperliquid was ~46% of all crypto buyback activity, averaging ~$65.5M/month, cumulative buybacks exceeding ~$1.3B. (Exact AF/HLP split is governance-adjustable and not officially published — treat percentages as estimates.)

**HLP vault.** A USDC vault running market-making strategies plus the Liquidator sub-vault, supplying USDC in Earn and accruing a share of fees. Liquidation PnL flows to depositors; no liquidation clearance fee, no HLP performance fee; most liquidations route to the open book so anyone can compete. This is the structural opposite of MEV accruing to searchers/validators. Returns of ~15–30% APR with ~5–12% drawdowns are third-party historical estimates that vary by window — verify against DefiLlama/HLP dashboard before publishing.

**Where MEV-like revenue accrues.** Validators capture **no order-flow MEV**: order-priority tips (IOC on HIP-3 assets, max 8 bps, paid in HYPE from undelegated stake, converted at spot mark) are **burned**, not paid to validators; a cancel-first rule guarantees cancels execute before all taker orders; HyperBFT instant single-slot finality removes reorg/backrun MEV. Liquidation surplus, spread, and funding capture are socialized into HLP for depositors; trading fees flow to HYPE holders via the Assistance Fund.

**Validator/staking economics — emissions, not fees.** Reward rate is inversely proportional to the square root of total HYPE staked (~2.37%/yr at 400M staked → ~2.2–2.4% in practice), paid from a future-emissions reserve, accruing per minute, distributed daily, auto-compounding. Validators need 10,000 HYPE self-stake (locked one year), may charge commission (cannot be raised unless new commission ≤1%), face a 7-day unstaking queue, and risk jailing (not automatic slashing) for poor performance. **Trading fees and MEV-like value do not flow to validators.**

## 7. Defense, Fairness & Real Incidents

**Structural MEV defenses.** (1) **No public mempool + deterministic price-time-priority matching** at the consensus layer removes the public observation window and discretionary intra-block reordering that enable AMM-style front-running/sandwiching — a protocol-level property, not an opt-in RPC/relay add-on. (2) **Stake-weighted validator median oracle** of off-chain CEX feeds means a single on-chain swap cannot move the reference price, blunting "manipulate-the-pool-then-liquidate" MEV. (3) **Robust multi-input mark price** means liquidations/stops fire on a multi-venue blend, not a single print. (4) **HyperBFT single-slot finality** kills reorg/backrun strategies.

**Residual risk surface (extraction moved, not removed).** Toxic-flow and self-liquidation risk is **socialized to HLP** rather than competed for by external bots; the safety mechanism is a **small validator quorum** that can override oracle prices and delist markets (effective against attackers but a censorship/trust concession); oracle integrity depends on honest validators and CEX feed availability — a risk Hyperliquid's own docs flag as **unmitigated**; and **HyperEVM** hosts a conventional gasPrice-priority MEV bot economy. Proposer ordering discretion *within* an action tier is the residual order-book trust assumption.

**Incident — Large ETH whale liquidation, March 12, 2025.** A trader deposited ~$15.23M USDC, used ~$4.3M as margin to open a 50x long of ~113,000 ETH (~$200M+ notional), then withdrew unrealized profit as ETH rose, pushing margin below maintenance and triggering liquidation into HLP. The trader walked away with ~$1.8–1.86M while **HLP absorbed a ~$4M loss**; HYPE fell ~8.5%. Not a smart-contract hack — exploitation of the margin/liquidation mechanic (offloading tail risk onto the vault). In response, Hyperliquid cut max leverage to 40x (BTC) / 25x (ETH) and raised maintenance margins on large positions. This exposed the HLP-as-backstop fragility weaponized two weeks later.

**Incident — JELLY oracle/short-squeeze, March 26, 2025 (canonical case).** An attacker opened a large JELLY short (reported ~$4.1M; combined short exposure cited up to ~$8M) alongside coordinated long wallets, pulled margin to self-liquidate so HLP/the Liquidator Vault inherited the short into a thin book, then pumped JELLY spot **~429%** (including via a Solana address) to distort the oracle. **HLP unrealized loss peaked at ~$12M (reported up to ~$13.5M)**; HYPE fell ~20%; Halborn cites ~$230M of HLP at risk had the price kept rising. Validators voted (consensus in ~2 minutes; settlement ~15:15 UTC) to **delist and force-settle 392M JELLY at the short's opening price $0.0095** — overriding the live oracle reading ~$0.50 — converting a potential eight-figure loss into a ~$703K HLP profit (Lookonchain). The Hyper Foundation pledged to make non-flagged users whole automatically from on-chain data. Attacker reportedly deposited ~$7.17M, withdrew ~$6.26M, ~$900K frozen. Bitget CEO Gracy Chen publicly criticized the handling ("FTX 2.0"). *Cite the short notional as a range ($4.1M–$8M).*

**Net assessment.** HyperCore's no-mempool + price-time-priority matching + median oracle genuinely removes the AMM-style sandwich/front-run vector for the core order book. But the residual risk *moved* rather than vanished: extraction migrates to who can self-liquidate into HLP and to HyperEVM gasPrice-priority arbitrage; the safety mechanism is a small validator quorum that can override/delist (effective but centralizing); and oracle integrity depends on honest validators and CEX feeds (docs flag prolonged oracle manipulation as unmitigated). Hyperliquid trades the permissionless-but-extractive MEV model for **protocol-controlled-but-centralized order flow**.

## 8. Comparison Table: Hyperliquid vs Solana + Jito

| Dimension | Hyperliquid (HyperCore) | Solana + Jito |
|---|---|---|
| **Public mempool** | None for trading orders; actions sequenced by HyperBFT before execution. (HyperEVM has a normal gasPrice mempool.) | No global public p2p mempool either; txs stream to the current/upcoming leader via Gulf Stream — leader-side visibility is the front-running vector. |
| **Transaction ordering** | Deterministic, protocol-fixed: per-block action categories (non-GTC/IOC → cancels → GTC/IOC), then price-time priority; proposer discretion only *within* a tier. | Leader has broad discretion over inclusion and ordering within its slot; Proof of History timestamps but does not constrain inclusion. |
| **MEV auction mechanism** | None Jito-style. Native priority fees (April 13, 2026) exist but are **burned**, in-protocol, IOC-on-HIP-3-only, 8 bps cap, cancels always first. | Off-chain Jito block engine runs sealed-bid bundle auctions; priority fees (lamports/CU) + Jito tips form a de-facto priority-gas-auction. |
| **Dominant MEV types** | Backstop liquidations (internalized to HLP); cross-venue HL↔CEX funding/basis arb; oracle/funding arb on thin perps; HyperEVM gasPrice arb. | Sandwiching, atomic arbitrage, lending/perp liquidation races, JIT-LP extraction. |
| **Who captures MEV** | Liquidation/spread/funding value socialized to **HLP depositors** and HYPE holders (via buybacks); cross-venue arb to external quant desks; HyperEVM arb to bots. Validators capture none. | External searcher/validator market: searchers, validators, and stakers via tips (Jito takes ~5%); liquidator bonuses to fastest bots. |
| **Latency game** | Fixed target: be in AWS Tokyo (ap-northeast-1), ~2–3 ms vs >200 ms. But ~880 ms of ~884 ms order-to-fill is consensus/server time, so colocation trims only the small network tail. | Moving target: colocate near the rotating upcoming leader and Jito relay/block-engine endpoints; sub-slot races on ~400 ms slots. |
| **Decentralization / trust** | Small set (~21–24 active, mid-2026; →27 planned), all in one AWS region; validators run oracle and can vote to override/delist (JELLY: ~2-min consensus). Centralized but extraction-resistant. | ~1,300+ globally distributed validators; rotating leader schedule. Decentralized but extraction-permissive. |
| **Sandwiching feasibility** | Structurally foreclosed on the CLOB (no public pending-order book, no gas auction, deterministic price-time priority). Feasible on the separate HyperEVM layer. | Feasible: leader-side tx visibility + priority-fee bidding + Jito bundle placement let searchers bracket victim orders. |

*Bottom line: the honest contrast is **discretionary/auctioned MEV (Solana + Jito) vs deterministic ordering with residual latency + cross-venue MEV (Hyperliquid)** — not "MEV vs no MEV." Hyperliquid behaves like a sealed, deterministic CLOB that eliminates the discretionary/auction MEV economy and socializes the largest internalized stream (liquidations) to HLP, while substituting governance/oracle centralization risk; Solana + Jito is an open auction for block space and ordering that pushes value to the edges.*
